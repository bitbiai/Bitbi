# AI Folders & Images Manual Review State Schema Design

Last updated: 2026-05-17

Phase 6.12 designed the manual-review state schema for AI folders/images owner-map issues. Phase 6.13 adds the additive schema foundation in `0057_add_ai_asset_manual_review_state.sql` only. Phase 6.14 adds local-only review-item import dry-run planning. Phase 6.15 adds an admin-approved import executor that can create only manual-review items/events and defaults to dry-run. Phase 6.16 adds read-only queue/evidence APIs for imported review rows. Phase 6.17 adds an admin-approved status workflow that updates only review item status fields and appends review events. Phase 6.18 adds status operator evidence rollups, Admin Control Plane visibility, API client wrappers, and review-status controls that operate only on review-state rows. These phases do not update ownership metadata, backfill rows, switch access checks, list/move/delete R2 objects, call providers, call Stripe, call Cloudflare APIs, mutate credits or billing, claim tenant isolation, or claim production readiness.

## Purpose

Phase 6.10 real main evidence showed that AI folders/images are not ready for ownership-metadata access switching or old-row ownership backfill:

| Signal | Count |
| --- | ---: |
| Folders scanned | 16 |
| Images scanned | 63 |
| Metadata missing total | 75 |
| Public unsafe | 21 |
| Derivative ownership risks | 63 |
| Simulated dual-read unsafe | 42 |
| Manual review needed | 90 |
| Metadata conflicts | 0 |
| Relationship conflicts | 0 |
| Orphan folder references | 0 |
| Organization-owned rows | 0 |

Phase 6.11 turned those signals into review categories and statuses. Phase 6.12 defined how an additive schema can persist operator review state, and Phase 6.13 adds the empty tables/indexes without performing review execution or ownership changes.

## Phase 6.13 Migration

Phase name: `Phase 6.13 - Additive Manual Review State Schema for AI Folders & Images`

Migration filename: `0057_add_ai_asset_manual_review_state.sql`

The migration is additive only. It creates the review-state tables and indexes, but it does not import evidence, create review rows, mutate existing folder/image rows, backfill ownership metadata, or change runtime access behavior.

## Phase 6.13 Tables

### `ai_asset_manual_review_items`

Purpose: one durable operator-review item per active owner-map issue.

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | TEXT PRIMARY KEY | Stable review item id generated by future import/admin code. |
| `asset_domain` | TEXT NOT NULL | `ai_folders`, `ai_images`, `relationship`, `public_gallery`, or `derivative`. |
| `asset_id` | TEXT | Safe asset id when the issue is row-specific. |
| `related_asset_id` | TEXT | Related folder/image/parent id for relationship and derivative issues. |
| `source_table` | TEXT | Source table, such as `ai_folders` or `ai_images`. |
| `source_row_id` | TEXT | Source row id when available. |
| `issue_category` | TEXT NOT NULL | One of the Phase 6.11 review categories. |
| `review_status` | TEXT NOT NULL | One of the Phase 6.11 review statuses. |
| `severity` | TEXT NOT NULL | `info`, `warning`, or `critical`. |
| `priority` | TEXT NOT NULL | `low`, `medium`, `high`, or `urgent`. |
| `legacy_owner_user_id` | TEXT | Legacy `user_id` signal, if safe and needed for review. |
| `proposed_asset_owner_type` | TEXT | Future reviewed owner class; not applied to source rows by this table. |
| `proposed_owning_user_id` | TEXT | Future reviewed owning user id, if approved. |
| `proposed_owning_organization_id` | TEXT | Future reviewed organization id; requires strong evidence. |
| `proposed_ownership_status` | TEXT | Future reviewed ownership status. |
| `proposed_ownership_source` | TEXT | Future reviewed source, such as `manual_review`. |
| `proposed_ownership_confidence` | TEXT | Future reviewed confidence value. |
| `evidence_source_path` | TEXT | Sanitized evidence source path or archive reference. |
| `evidence_report_generated_at` | TEXT | Timestamp from the evidence report. |
| `evidence_summary_json` | TEXT | Bounded safe summary only; no raw prompts, keys, tokens, or provider payloads. |
| `safe_notes` | TEXT | Operator notes after sanitization and length limits. |
| `assigned_to_user_id` | TEXT | Optional reviewer assignment. |
| `reviewed_by_user_id` | TEXT | Reviewer who reached the current terminal status. |
| `reviewed_at` | TEXT | Review completion timestamp. |
| `created_by_user_id` | TEXT | Actor or importer that created the review item. |
| `created_at` | TEXT NOT NULL | Creation timestamp. |
| `updated_at` | TEXT NOT NULL | Last update timestamp. |
| `superseded_by_id` | TEXT | Replacement review item id when newer evidence supersedes this item. |
| `metadata_json` | TEXT | Bounded safe metadata for UI labels and import fingerprints only. |

Design constraints:

- Deterministic review-item dedupe is deferred to future import logic.
- Allowed categories, statuses, severities, priorities, owner classes, and confidence values should be enforced in application code and tested before any write endpoint exists.
- `evidence_summary_json`, `safe_notes`, and `metadata_json` must be length-bounded and sanitized.
- The table records review intent only. It must not be treated as applied ownership metadata.

### `ai_asset_manual_review_events`

Purpose: append-only audit trail for review item creation, assignment, note, and status changes.

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | TEXT PRIMARY KEY | Stable event id. |
| `review_item_id` | TEXT NOT NULL | Parent review item id. |
| `event_type` | TEXT NOT NULL | One of the allowed future review actions. |
| `old_status` | TEXT | Status before the event. |
| `new_status` | TEXT | Status after the event. |
| `actor_user_id` | TEXT | Admin/reviewer id. |
| `actor_email` | TEXT | Optional safe admin email for audit display. |
| `reason` | TEXT | Bounded reason. Required for status changes. |
| `idempotency_key` | TEXT | Hashed or bounded operator-supplied idempotency signal; raw values should not be exposed. |
| `request_hash` | TEXT | Safe request fingerprint, never raw request body. |
| `event_metadata_json` | TEXT | Bounded safe event metadata. |
| `created_at` | TEXT NOT NULL | Event timestamp. |

Design constraints:

- Events are append-only.
- Events must not contain raw prompt/provider/R2/secret/payment data.
- Status changes must create an event.
- Future write APIs must require `Idempotency-Key`; duplicate same-key/same-request writes return the same result, while same-key/different-request conflicts fail.

## Phase 6.13 Indexes

The migration creates these non-unique indexes:

| Index | Target columns | Purpose |
| --- | --- | --- |
| `idx_ai_asset_manual_review_items_domain_asset` | `asset_domain`, `asset_id` | Find active issues for one row/domain. |
| `idx_ai_asset_manual_review_items_status` | `review_status` | Review queues by status. |
| `idx_ai_asset_manual_review_items_category` | `issue_category` | Category-specific queues. |
| `idx_ai_asset_manual_review_items_severity` | `severity` | Highest-risk queues first. |
| `idx_ai_asset_manual_review_items_priority` | `priority` | Operator work ordering. |
| `idx_ai_asset_manual_review_items_created_at` | `created_at` | Audit and pagination. |
| `idx_ai_asset_manual_review_items_evidence_source` | `evidence_source_path` | Trace items back to evidence. |
| `idx_ai_asset_manual_review_events_item` | `review_item_id`, `created_at` | Event history for one review item. |
| `idx_ai_asset_manual_review_events_idempotency` | `idempotency_key` | Future write idempotency. |

A unique dedupe index on nullable asset fields is deferred because SQLite/D1 nullable uniqueness semantics could allow duplicate active items. Future import code should use deterministic ids or application-level dedupe before any row import is approved.

## State Transition Model

Allowed future transitions:

| From | To | Requirement |
| --- | --- | --- |
| `pending_review` | `review_in_progress` | Reviewer assignment or explicit start action. |
| `review_in_progress` | approved statuses | Bounded reason and safe evidence summary. |
| `review_in_progress` | blocked statuses | Bounded reason and category-specific risk. |
| `review_in_progress` | `needs_legal_privacy_review` | Legal/privacy risk detected. |
| `review_in_progress` | `deferred` | Bounded reason. |
| `deferred` | `pending_review` | Bounded reason and updated priority. |
| `rejected` | `pending_review` | Bounded reason and new evidence. |
| approved or blocked statuses | `superseded` | New evidence replaces the item. |
| any active status | `needs_legal_privacy_review` | Legal/privacy risk detected. |

Forbidden future transitions:

- blocked directly to approved without a review reason and event;
- approved directly to backfill execution;
- any status to access-switch approval;
- deletion of review history;
- status changes without event rows;
- status changes that include unsafe raw evidence fields.

## Review Item Identity And Deduplication

Future imports from evidence should use deterministic keys:

| Issue type | Review item key input |
| --- | --- |
| Folder/image row issue | `asset_domain + asset_id + issue_category` |
| Folder/image relationship issue | `relationship + asset_id + related_asset_id + issue_category` |
| Public gallery issue | `public_gallery + asset_id + issue_category` |
| Derivative issue | `derivative + parent_image_id + issue_category` |

Rules:

- repeated imports from the same or later evidence must not create duplicate active items;
- if evidence still shows the issue, add an event only when status/category/severity changes;
- if evidence no longer shows the issue, a future phase may mark the item `superseded` or `deferred`, never delete history;
- if category or severity changes, add an event rather than destructively rewriting history;
- evidence import must be idempotent and bounded.

## Audit Model

Every future write should create one `ai_asset_manual_review_events` row. Event metadata should include:

- event type;
- old/new status;
- actor id/email if available;
- bounded reason;
- evidence source path;
- safe evidence timestamp;
- idempotency outcome;
- correlation id if available.

Events must not store raw prompts, provider bodies, private R2 keys, signed URLs, cookies, auth headers, Stripe data, Cloudflare tokens, private keys, raw idempotency keys, raw request fingerprints, or unsafe full metadata blobs.

## API Design

Phase 6.15 implements the import endpoint, Phase 6.16 implements read-only queue/evidence endpoints, and Phase 6.17 implements the bounded status endpoint:

- `POST /api/admin/tenant-assets/folders-images/manual-review/import`
- `GET /api/admin/tenant-assets/folders-images/manual-review/items`
- `GET /api/admin/tenant-assets/folders-images/manual-review/items/:id`
- `GET /api/admin/tenant-assets/folders-images/manual-review/items/:id/events`
- `POST /api/admin/tenant-assets/folders-images/manual-review/items/:id/status`
- `GET /api/admin/tenant-assets/folders-images/manual-review/evidence`
- `GET /api/admin/tenant-assets/folders-images/manual-review/evidence/export`

The read endpoints are admin-only, production-MFA protected, bounded, sanitized, and read-only. They expose queue filters, item detail, event history, rollups, and JSON/Markdown evidence export. The status endpoint is admin-only, production-MFA protected through route policy, same-origin protected, rate-limited, idempotency-guarded, and requires `confirm: true` plus a bounded `reason`. It can update only `ai_asset_manual_review_items.review_status`/review metadata and create a matching `ai_asset_manual_review_events` row. None of these endpoints backfill ownership, switch access checks, mutate source asset rows, create notes, or touch R2.

Possible future endpoints, not implemented in Phase 6.17:

| Endpoint | Method | Purpose | Requirements |
| --- | --- | --- | --- |
| `/api/admin/tenant-assets/folders-images/manual-review-items/import-from-evidence` | POST | Future bounded import from approved evidence. | Admin-only, production MFA, same-origin JSON, `Idempotency-Key`, no backfill. |
| `/api/admin/tenant-assets/folders-images/manual-review-items/:id/note` | POST | Future bounded note event. | Admin-only, production MFA, same-origin JSON, `Idempotency-Key`, event audit. |

Future endpoints must not apply ownership changes, backfill rows, switch access checks, list/mutate R2, call providers, call Stripe, mutate Cloudflare, change public visibility, or change credits/billing.
