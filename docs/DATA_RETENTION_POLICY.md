# Data Retention Policy Engineering Baseline

Date: 2026-04-26

Last reconciled: 2026-05-15

This file is an engineering baseline for retention and deletion behavior. It is not legal advice and does not claim GDPR/CCPA compliance. Product/legal review is required before enabling irreversible deletion or promising user-facing retention terms.

Current release truth: `config/release-compat.json` declares the latest auth D1 migration as `0048_add_member_ai_usage_attempts.sql`. This policy baseline does not approve production deploy, full live billing readiness, full SaaS maturity, full tenant isolation, or full legal compliance.

## Principles

- Keep authentication secrets, session material, MFA material, reset tokens, service signatures, and provider credentials out of exports.
- Preserve security/audit evidence unless a reviewed retention policy explicitly allows deletion or anonymization.
- Prefer dry-run deletion plans and admin approval before any irreversible operation.
- Treat generated AI prompts, previews, media, wallet addresses, contact messages, and profile fields as potentially personal data.
- Do not inline large R2 binaries into export archives; Phase 1-I export archives use authorized JSON manifests with media references only and do not expose raw internal media R2 keys.
- Delete only owner-scoped or lifecycle-scoped R2 objects whose prefix has been explicitly approved. Phase 1-J cleanup is limited to `AUDIT_ARCHIVE` objects under `data-exports/`.
- Treat Admin Control Plane, Pricing page, and Credits dashboard data as presentation surfaces over existing backend records unless a future phase adds new storage. They must not create separate retention promises.

## Current Enforcement

Phase 1-H enforces request creation, planning, approval state, idempotency, route policy registration, admin-only access, fail-closed rate limiting, same-origin checks, and byte-limited bodies. Phase 1-I adds bounded export archive generation and authorized admin archive download for approved export plans. Phase 1-J adds bounded expired-archive R2 cleanup and a safe executor pilot for reversible auth-state cleanup. Phase 2-A adds organization/membership storage. Phase 2-B adds billing/entitlement/credit-ledger storage and admin credit-grant APIs. Phase 2-C adds opt-in org-scoped image generation credit debits. Phase 2-D adds opt-in org-scoped image-generation usage attempts/reservations. Phase 2-E adds bounded cleanup for expired/stuck usage attempts, releases stale reservations without debits, expires replay metadata after its replay window, and adds sanitized admin inspection. Phase 2-F deletes only expired attempt-linked temporary replay objects after strict prefix/user/attempt validation. Phase 2-H adds backend-only org-scoped text generation credit debits and bounded text replay metadata. Phase 2-I adds provider-neutral billing event ingestion with raw-body synthetic verification, idempotent event storage, dry-run action planning, and sanitized admin inspection. Phase 2-J adds Stripe Testmode credit-pack checkout session tracking and verified Testmode checkout credit grants. Phase 2-K locks Testmode checkout to platform admins. Phase 2-L adds narrow live one-time credit-pack checkout and a gated Credits dashboard, backed by additive checkout authorization/payment-state metadata in migration `0040`. Phase 2-M adds no new storage; it reuses existing billing and attempt tables for platform-admin-only BFL Admin AI image-test charges. Later migrations add member credit ledgers/live checkout (`0041`/`0042`), News Pulse/OpenClaw storage (`0043`-`0045`), asset storage quota rows (`0046`), member subscription/credit bucket rows (`0047`), and member personal AI usage attempts (`0048`) now used for member image, member music, and member video gateway attempts plus Phase 3.7 replay-unavailable, cover-status, finalization, and cleanup metadata. Phase 4.1 adds admin/platform AI budget policy design, registry metadata, and baseline/check metadata only. Phase 4.2 adds pure admin/platform budget policy helper contracts, deterministic tests, and stricter baseline/check metadata only. Phase 4.1/4.2 add no data storage, no runtime budget enforcement, no route behavior, no provider calls, no credit mutations, and no Admin UI behavior. The Pricing/Credits/BITBI Pro/account AI surfaces add no separate page storage but depend on migrations through `0048` for the current auth Worker behavior; the Admin Control Plane adds no new storage and only renders sanitized existing API/readiness data. Org, billing, member subscription, credit bucket, member AI usage attempt, storage quota, News Pulse, and OpenClaw data are not yet part of export/delete planning. The system still does not execute irreversible deletion or hard-delete user/media/audit records by default.

## Contact Processor Policy Gap

Contact form content is sent through Resend and is not stored in a repo-owned D1 table. Until a processor workflow or repo-owned contact-message store exists:

- Contact messages are treated as high-PII.
- Export/delete requests involving contact messages require manual processor/mailbox review.
- No repository API currently promises contact-message export or deletion.
- Product/legal must define the Resend/mailbox retention window before compliance claims are made.

## Historical R2 Ownership Policy

Phase 1-J cleanup is intentionally narrow. It only deletes `AUDIT_ARCHIVE` objects whose keys match the generated export archive prefix. Do not enable deletion for these categories until ownership is proven by D1 rows or a dry-run owner-map backfill:

- Audit/activity archive JSONL chunks outside `data-exports/`.
- Legacy `PRIVATE_MEDIA` keys not linked to `profiles`.
- Legacy `USER_IMAGES` keys not linked to `ai_images`, `ai_text_assets`, or `ai_video_jobs`.
- Any provider/transient object without an owner row.
- Stripe Testmode/live checkout, billing-event, credit-ledger, and usage-event records unless a legal/product-approved billing retention policy allows deletion or anonymization.

## Required Before Destructive Deletion

1. Legal/product approval of retention windows and deletion exceptions.
2. Staging verification that every R2 object key pattern is owner-scoped and complete.
3. A two-step approval/execution model with idempotency and audit events.
4. A recovery grace period or explicit no-recovery policy.
5. Dry-run comparison of planned D1/R2 actions against expected user data.
6. Tests proving cross-user data is not included and audit records are retained/anonymized correctly.
7. Runbook updates for accidental deletion, rollback limitations, and support communication.
