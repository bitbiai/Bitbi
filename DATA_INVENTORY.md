## Summary

The product stores user account data, authentication state, wallet addresses, profile/avatar data, favorites, generated AI assets, admin/user activity, async video job metadata, poison-message diagnostics, and operational cleanup state in Cloudflare D1 and R2. The contact Worker sends contact form content through Resend; this repository does not currently define durable contact-message storage.

Phase 1-H adds request tracking tables for export/deletion/anonymization planning. Phase 1-I adds bounded JSON export archive generation into private R2 for approved export plans. Phase 1-J adds bounded cleanup for expired export archive objects and a safe executor pilot for reversible auth-state cleanup. Phase 2-A adds organization and membership foundation tables but does not migrate existing user-owned assets into tenant ownership. Phase 2-B adds billing/plan/entitlement/credit ledger foundation tables without enabling broad live payment readiness. Phase 2-C starts writing organization-scoped `usage_events` / `credit_ledger` debits only for `/api/ai/generate-image` requests that explicitly include organization context. Phase 2-D adds `ai_usage_attempts` so those org-scoped image requests can reserve credits, avoid duplicate provider calls on idempotent retries, and replay stored temporary results when available. Phase 2-E adds bounded cleanup/admin inspection for expired or stuck `ai_usage_attempts`. Phase 2-F adds bounded, prefix-scoped cleanup for expired temporary replay objects after validating attempt linkage. Phase 2-G was an inspection/no-op because no safe existing member-facing text generation route existed. Phase 2-H adds org-scoped member text generation at `/api/ai/generate-text`, using the same `usage_events`, `credit_ledger`, and `ai_usage_attempts` foundation with bounded text replay metadata in `metadata_json`; it does not add no-org text generation or charge admin/text-asset routes. Phase 2-I adds provider-neutral billing event ingestion tables and a synthetic test webhook foundation; it stores payload hashes and sanitized summaries only, does not store raw webhook bodies/signatures/payment method data, and does not grant credits or update subscriptions from provider events. Phase 2-J adds Stripe Testmode checkout session tracking and verified Testmode checkout credit grants for server-side credit packs only. Phase 2-K locks Testmode checkout to platform admins. Phase 2-L adds narrow live Stripe one-time credit-pack checkout for platform admins and active organization owners, with migration `0040` adding checkout authorization/payment-state fields. Phase 2-M updates the live credit-pack catalog to 5,000 credits / 9.99 EUR and 12,000 credits / 19.99 EUR, and charges existing organization credits when platform admins run Admin AI image tests for supported BFL models. Later migrations add member credit ledgers and live checkout rows (`0041`/`0042`), News Pulse/OpenClaw storage (`0043`-`0045`), per-user asset storage quota tracking (`0046`), member subscription plus credit bucket rows (`0047`), member personal AI usage attempts (`0048`) used by the Phase 3.4 image gateway pilot, Phase 3.6 member music gateway migration, Phase 3.7 replay-unavailable/cover-status/finalization/cleanup hardening, and Phase 3.8 member video gateway migration, admin video job budget metadata columns (`0049`) used only by Phase 4.5, News Pulse visual budget metadata columns (`0050`) used only by Phase 4.6, and admin text/embeddings/music/compare/live-agent idempotency attempts (`0051`) used by Phase 4.8.1, inspected/cleaned by Phase 4.8.2, extended to Admin Music by Phase 4.9, extended to Admin Compare by Phase 4.10, and extended to Admin Live-Agent stream sessions by Phase 4.12. Phase 4.1 adds admin/platform AI budget policy design and metadata/check updates only; Phase 4.2 adds pure admin/platform budget policy helper contracts, deterministic tests, and stricter baseline/check metadata only; Phase 4.3 adds safe budget-policy plan/audit metadata to the existing charged Admin BFL image-test response, `usage_events`, and `ai_usage_attempts` metadata; Phase 4.4 adds read-only Admin/Platform AI budget evidence reporting; Phase 4.5 adds sanitized `platform_admin_lab_budget` metadata to admin async video job rows and bounded queue messages before internal video task create/poll; Phase 4.6 adds sanitized `openclaw_news_pulse_budget` metadata to News Pulse visual rows before generated-thumbnail provider calls. Phase 4.7 adds no new storage or migration; it defines a signed internal caller-policy body key, validates/sanitizes it in the AI Worker after service-auth, strips it before provider payload handling, and propagates safe metadata only for covered caller paths where feasible. Phase 4.8 adds no new storage or migration; admin text and embeddings test routes require `Idempotency-Key`, build safe `platform_admin_lab_budget` metadata in-memory, return sanitized admin-only budget/caller summaries, and send signed `budget_metadata_only` caller-policy metadata to the AI Worker. Phase 4.8.1 adds `admin_ai_usage_attempts` rows for those two routes; rows store hashed idempotency keys, request fingerprints, sanitized budget/caller-policy metadata, status, and safe result summaries, but not raw prompts, raw embedding input, generated text, embedding vectors, provider request bodies, cookies, auth headers, Stripe data, Cloudflare tokens, secrets, or private keys. Phase 4.8.2 adds no new storage or migration; it adds admin-only sanitized list/detail APIs, a dry-run-default bounded cleanup endpoint, and scheduled bounded cleanup that marks expired pending/running admin AI attempts without deleting rows. Phase 4.9 adds no new storage or migration; it extends those sanitized metadata-only rows to Admin Music and does not store raw prompts, lyrics, audio, or provider bodies. Phase 4.10 adds no new storage or migration; it extends those sanitized metadata-only rows to Admin Compare and does not store raw prompts, compare outputs, provider request bodies, or provider response bodies. Phase 4.11 adds no new storage or migration; it audits Admin Live-Agent only. Phase 4.12 adds no new storage or migration; it extends those sanitized metadata-only rows only to Admin Live-Agent stream sessions and does not store raw messages, streamed output, provider request bodies, or provider response bodies. Phase 4.13 adds no new storage or migration; it classifies sync video debug as retired/disabled-by-default and keeps async admin video jobs as the supported budgeted admin video path. Phase 4.6/4.7/4.8/4.8.1/4.8.2/4.9/4.10/4.11/4.12/4.13 add no real provider calls by tests/Codex, public billing behavior, Admin UI behavior, Stripe calls, credit mutation, runtime env kill-switch enforcement, broad admin/platform/internal route migration, Admin video changes beyond Phase 4.5, OpenClaw/News Pulse changes beyond Phase 4.6 compatibility, platform/background AI migration outside News Pulse visuals, org-scoped member route behavior changes, member image/music/video billing changes, or global hard-fail unrelated internal AI Worker migration. These additions do not prove invoices, customer portal, Stripe Tax, refunds/disputes/chargebacks, reconciliation operations, public production billing, full subscription lifecycle readiness, or full AI cost-gateway coverage. The Admin Control Plane is a frontend/admin inspection surface over existing sanitized APIs and does not add new storage. Cleanup still does not hard-delete attempts, ledger rows, usage events, saved media, private media, audit archives, export archives, video outputs, derivatives, billing events, checkout sessions, subscription rows, credit bucket rows, member AI usage attempts, admin AI usage attempts, admin video job rows, News Pulse rows, or unrelated R2 data. The current lifecycle system does not execute irreversible deletion by default and does not inline binary R2 media into exports.

Excluded from destructive deletion until owner mapping is proven:

- Audit/activity archive JSONL chunks outside `data-exports/`.
- Any legacy/private media key not linked to a D1 owner row.
- Any provider/transient object without a D1 owner row.
