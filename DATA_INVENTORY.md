## Summary

The product stores user account data, authentication state, wallet addresses, profile/avatar data, favorites, generated AI assets, admin/user activity, async video job metadata, poison-message diagnostics, and operational cleanup state in Cloudflare D1 and R2. The contact Worker sends contact form content through Resend; this repository does not currently define durable contact-message storage.

Phase 1-H adds request tracking tables for export/deletion/anonymization planning. Phase 1-I adds bounded JSON export archive generation into private R2 for approved export plans. Phase 1-J adds bounded cleanup for expired export archive objects and a safe executor pilot for reversible auth-state cleanup. Phase 2-A adds organization and membership foundation tables but does not migrate existing user-owned assets into tenant ownership. Later phases add billing, member AI gateway attempts, admin/platform AI budget evidence, app switches, cap/repair/report/archive tables through `0055`, Phase 6.4 nullable folder/image owner metadata in `0056`, and Admin Control Plane inspection surfaces. Phase 6.1-6.9 are tenant asset migration foundations: they document the target owner model, add source/fixture dry-run inventory/tests for folders/images, define the future schema/access impact plan, add nullable columns, write metadata for new personal folder/image rows only, simulate dual-read diagnostics, expose bounded admin evidence/report export, add operator evidence collection docs, and create a main-only pending evidence package, but no route access, lifecycle, R2, generation, quota, old-row ownership backfill, org ownership assignment, or billing behavior changes. Cleanup still does not hard-delete attempts, ledger rows, usage events, saved media, private media, audit archives, export archives, video outputs, derivatives, billing events, checkout sessions, subscription rows, credit bucket rows, member/admin AI attempts, admin video job rows, News Pulse rows, platform budget rows, tenant ownership rows, or unrelated R2 data. The current lifecycle system does not execute irreversible deletion by default and does not inline binary R2 media into exports.

## Historical R2 Ownership Policy

Known owner-linked prefixes:

- `PRIVATE_MEDIA` avatar keys under `avatars/{userId}`.
- `USER_IMAGES` generated media keys under `users/{userId}/...`.
- `AUDIT_ARCHIVE` data export archive keys under `data-exports/{subjectUserId}/{requestId}/{archiveId}.json`.
- `AUDIT_ARCHIVE` platform budget evidence archive keys under `platform-budget-evidence/platform_admin_lab_budget/{archiveId}.json` or `.md`.

Excluded from destructive deletion until owner mapping is proven:

- Audit/activity archive JSONL chunks outside `data-exports/`.
- Any legacy/private media key not linked to a D1 owner row.
- Any provider/transient object without a D1 owner row.
